前面几章带大家看完了一个"相对完整"的 Phi。但"相对完整"这几个字要打个引号——Phi 自己有一份文档,
专门记录"哪些真实 Agent 需要、但我们判断值不值得现在做"的清单。这一章不只是罗列这份清单,而是从
"你已经用 Agent 做过的一件事"出发:如果规模变大、场景变复杂,今天课上演示过的这个 Harness 会在哪
里先撑不住,你会怎么改?这可能是全课程最重要的一课:一个成熟的工程项目,边界是主动划出来的,不是能
力不够、划不到那里去——但划在哪里,永远值得被继续追问。

每一节都是"一个具体场景 → Phi 现在的选择 → 一道思考题"。思考题没有标准答案,折叠起来的是一种可能
的方向,不是唯一解;真正的讨论应该发生在课堂上。

## 执行隔离

**场景**:你把 Agent 接进了日常开发流程,让它自己跑测试、装依赖、跑构建脚本。有一次它为了"修好一
个失败的测试",判断需要清理一份看起来像缓存的目录,生成了一条 `bash` 命令。这条命令本身没有恶
意,但工作区之外恰好挂着一个符号链接指向你机器上另一个项目——`ConfinedFileSystem` 管得住文件工
具,但 `bash` 从来都是 `UNCONFINED`,这条命令照单全收地跑完了。

Phi 现在的选择:`ConfinedEnvironment` 只在进程内做路径检查——"代码写得规矩就不会越界",不是操作
系统级别的强制隔离。真正的沙箱(macOS Seatbelt、Linux Landlock/seccomp)能在工具本身有 bug 或者
被误导的情况下依然兜底,但要跨平台各做一套,维护成本完全是另一个量级;而且即便加了沙箱,`bash`、
MCP 工具这些"不受限"的能力依然不受它保护。`deferred.md` 给出的触发条件是:**当 Phi 必须在比
"进程内策略"更强的保证下执行不可信命令时**,才值得引入操作系统级沙箱。

**思考**:在真正上操作系统沙箱这种重工程之前,你会先加哪个更轻量的机制,把上面那次事故的发生概率
降到可接受的程度?

??? success "一种可能的思路"

    可以从"事前提醒"和"事后可退"两个方向找便宜的中间方案:执行前对匹配到常见危险模式(递归删
    除、改变权限、访问工作区外绝对路径)的命令单独提高审批门槛,即使当前处于 `accept_edits` 这类
    宽松模式;工作区所在的文件系统换成支持快照/可回滚的方案,把"执行前拍个快照"变成低成本的默认
    动作。这两者都不需要跨平台沙箱工程,但能显著降低最坏情况的代价。

## 多智能体的成熟度

**场景**:你想用委派一次性重构十个模块,让父 Agent 给每个模块各 `spawn_agent` 一个子 Agent 并行
处理。这十个模块其实共享同一套新接口约定,但子 Agent 之间互相看不到对方的进展——两个子 Agent 各自
对接口做出了不同的假设,等父 Agent 检查结果时,才发现两份实现互相对不上,只能返工。

Phi 现在的选择:只有"父 Agent 派生子 Agent、可以查看、可以非破坏性地插一句话"这一种模式。真正成
熟的多智能体系统还会有平等节点之间互相通信、打断子 Agent 当前动作直接重启、父 Agent 界面上能实时
看到多个子 Agent 同时在干什么——这些每一条单独拿出来都够做成一个独立项目。

**思考**:不做"完全对等通信"这种大工程,你能想到哪个折中方案,能在这个具体场景里避免返工?

??? success "一种可能的思路"

    一个便宜的折中是:在 spawn 之前,先由父 Agent 生成一份简短的共享约束(比如接口签名、命名规
    范),作为每个子 Agent 委派任务的一部分注入,而不是让子 Agent 各自"猜";如果任务之间的关联程
    度高到共享约束都不够,退一步选择顺序委派而不是并行——牺牲一部分并行度,换取不需要引入子 Agent
    间通信这层复杂度。

## 长期记忆

**场景**:你几乎每天都用同一个 Agent 继续同一个项目,但每次开新 Session,都要重新告诉它"我们用什
么框架、命名规范是什么、上次为什么放弃了某个方案"。Session 保留了完整历史没错,但那是"这一条对话
线"的历史——换一次 Session、甚至同一条历史被 Compaction 掉之后,这些信息就得你自己重新讲一遍。

Phi 现在的选择:先分清楚 Session(完整对话)和 Context(投影出的一部分)都不是"记忆"。真正的长期
记忆要回答一串产品问题:这条记忆归谁所有、什么时候该写入、新旧信息冲突了听谁的、检索时怎么排序、
什么时候该被遗忘、要不要考虑隐私——这些需要一个真实的复用场景才能设计清楚,不是"先做一个通用记忆
系统,以后总用得上"。

**思考**:在真正设计一套通用记忆系统之前,能不能用 Phi 已有的机制(Project Instructions、
Skills)拼出一个覆盖大部分场景的轻量方案?

??? success "一种可能的思路"

    可以让 Project Instructions 里维护一份持续更新的"项目笔记"(技术栈、约定、放弃过的方案),
    由用户或者 Agent 在每次任务结束时显式追加,而不是期待系统自动判断"这条该不该记住"。这没有解
    决排序、遗忘、隐私这些完整记忆系统要处理的问题,但覆盖了"同一个项目反复用"这个最常见的场景,
    而且完全建立在已有机制上,不需要新增一套存储和检索。

## 工具生态规模化

**场景**:你把公司内部十几个 MCP server 都连了起来,算下来两三百个工具。你发现 Agent 越来越容易
选错工具,或者干脆没注意到任务里真正该用的那个——每次请求的工具 schema 已经占了很大一块 Context
预算,模型的"注意力"被摊得很薄。

Phi 现在的选择:直接把每个连上的 MCP 工具注册成一个具名工具。如果连接的服务器多到这个规模,更合理
的做法可能是只给模型两个元工具——"搜索有哪些工具"和"执行某个工具",背后用搜索索引动态挑选。Phi
判断现在还没连接到需要这么做的规模,所以选了更简单、更透明的直接注册。

**思考**:在真正做"搜索式发现层"这套工程之前,你会先试哪个更简单的中间方案,来缓解这个具体场景里
的问题?

??? success "一种可能的思路"

    可以先不动 Tool 层,而是在配置层面下功夫:按项目或任务显式声明这次要连接哪些 server,而不是一
    次性全连;或者给工具按 server/领域分组,只把当前任务大概率用得到的组注册进 Context,其余组留
    在"未连接"状态,需要时再显式打开。这不需要检索基础设施,只是把"要不要暴露这个工具"这个决定,
    从"运行时自动判断"退回成"配置时人工判断"。

## 执行模型与运营

**场景**:一次任务里,Agent 需要读五个互不相关的文件。今天的 Phi 会一个接一个地读,每次工具往返都
有网络和进程调度开销,你能明显感觉到等待时间比"如果这五个读取能一起跑"要长。月底你还发现,这个月
已经调用了很多次,但具体某一次任务花了多少钱,你完全没有实时的概念,只能事后去 Proxy 管理界面查。

Phi 现在的选择:一次模型回复里如果有多个工具调用,是顺序执行的,不是并行。让工具声明自己"并发安
全"、符合条件就并行跑的设计被考虑过,但没有采纳,因为一旦允许并行,失败顺序、取消语义、审批流程、
Event 顺序全部要重新定义一遍,复杂度不是"加一行 `asyncio.gather`"能打发的。费用方面,LiteLLM 只在
部分非流式响应里报出真实费用,流式场景拿不到,很多受限的 virtual key 也访问不了计费管理接口。

**思考**:并行执行和成本可见性这两个问题,有没有比"通用 `parallel_safe` 契约"和"等 Proxy 开放实时
计费接口"风险更小的第一步?

??? success "一种可能的思路"

    并行方面可以先从课程里已经学过的 `approval_class` 入手:只对同一响应里的多个 `read_only` 调
    用做并行,不去碰 `mutates_workspace` 和 `unconfined`——因为只读调用之间几乎不存在需要重新定
    义的失败顺序和取消语义,复杂度比通用契约小得多。成本方面可以先做一个本地粗略估算:用第 04 章
    学过的 token 估算公式乘以已知的模型单价,实时展示一个"预估花费",不追求和账单完全一致,只求让
    用户在任务进行中就有数量级概念。

## 讨论

以上每一节的"Phi 现在的选择"背后,`deferred.md` 里都写了一句"什么条件出现时应该重新考虑"(比如
"当已连接的工具 schema 明显撑爆了 Context 预算"再去做 MCP 的搜索式发现层)。选一条你觉得对你们团
队最贴近现实的,想一想:如果没有这条触发标准,只是单纯"觉得这个功能听起来很有用",直接现在就做,
会有什么风险?

??? success "展开参考答案"

    以 MCP 工具搜索为例:在工具数量还很少时提前增加搜索式发现层,会引入索引、排序、召回率、缓存和
    权限过滤等新问题,还可能因为缺少真实使用数据而选错抽象。结果是 Context 未必节省多少,系统却已
    经增加了维护成本、延迟和新的失败路径。

    "工具 schema 已经明显占用 Context 预算"这样的触发标准,把决定建立在可观察的问题上。到那时,
    我们可以根据真实的工具数量、token 占用和调用模式设计方案,而不是只因为功能听起来有用就提前建
    设。
