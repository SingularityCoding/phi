前面几章带大家看完了一个"相对完整"的 Phi。但"相对完整"这几个字要打个引号——Phi 自己有一份文档,
专门记录"哪些真实 Agent 需要、但我们判断值不值得现在做"的清单。这一章不看代码,看的是这份清单背后
的判断逻辑。这可能是全课程最重要的一课:一个成熟的工程项目,边界是主动划出来的,不是能力不够、划不
到那里去。

## 执行隔离

Phi 的 `ConfinedEnvironment` 只是在进程内做路径检查——本质上是"代码写得规矩,就不会越界",而不是操
作系统级别的强制隔离。真正的沙箱(macOS Seatbelt、Linux Landlock/seccomp)能在工具本身有 bug 的情
况下依然兜底,但要跨平台各做一套,维护成本完全是另一个量级。而且即便加了沙箱,`bash`、MCP 工具这些
"不受限"的能力依然不受它保护——沙箱不是万能药,只是把风险面从"整个进程"缩小到"部分工具"。

## 多智能体的成熟度

Phi 现在只有"父 Agent 派生子 Agent、可以查看、可以非破坏性地插一句话"这一种模式。真正成熟的多智能
体系统还会有:平等节点之间互相通信(而不只是父子关系)、打断子 Agent 当前动作直接重启(而不是等到
下一步边界才插话)、父 Agent 界面上能实时看到多个子 Agent 同时在干什么(而不是事后看汇总状态)。这
些每一条单独拿出来都够做成一个独立的项目。

## 长期记忆

这大概是目前业界讨论最多的话题。先要弄清楚一件事:Session 保存了完整对话,Context 从里面投影出一部
分喂给模型——这两个都不是"记忆"。真正的长期记忆要回答一串产品问题,而不只是技术问题:这条记忆归谁
所有、什么时候该写入、新旧信息冲突了听谁的、检索的时候怎么排序、什么时候该被遗忘、要不要考虑隐私。
Phi 判断这些问题需要一个真实的复用场景才能设计清楚,不是"先做一个通用记忆系统,以后总用得上"。

## 工具生态规模化

今天 mini-agent 只有 3 个工具,Phi 直接把每个连上的 MCP 工具注册成一个具名工具。如果连接的服务器多
到几十上百个,所有工具的 schema 全部塞进每次请求,会把模型能看到的"工具预算"直接撑爆——这时候更合
理的做法可能是只给模型两个元工具:"搜索有哪些工具"和"执行某个工具",背后用搜索索引去动态挑选。Phi
判断现在还没连接到需要这么做的规模,所以选了更简单、更透明的直接注册。同一类问题还包括:MCP 目前只
支持本地 stdio 服务器,不支持需要 OAuth 的远程服务;MCP 参数校验完全信任远程服务器返回的错误,没有
在本地再做一层 JSON Schema 校验。

## 执行模型与运营

Phi v1 里,一次模型回复里如果有多个工具调用,是**顺序**执行的,不是并行。想让工具声明自己"并发安
全"、符合条件就用 `asyncio.gather` 并行跑——这个设计被考虑过,但没有采纳,因为一旦允许并行,失败
顺序、取消语义、审批流程、Event 顺序全部要重新定义一遍,复杂度不是"加一行 gather"能打发的。另一个
经常被提到的运营需求是"这次调用花了多少钱"——技术上 LiteLLM 能在部分非流式响应里报出真实费用,但
流式场景拿不到,而且很多受限的 virtual key 也访问不了计费管理接口,所以现在只能让用户自己去 Proxy
的管理界面查。

## 讨论

以上每一条,`deferred.md` 里都写了一句"什么条件出现时应该重新考虑"(比如"当已连接的工具 schema 明
显撑爆了 Context 预算"再去做 MCP 的搜索式发现层)。选一条你觉得对你们团队最贴近现实的,想一想:如
果没有这条触发标准,只是单纯"觉得这个功能听起来很有用",直接现在就做,会有什么风险?

??? success "展开参考答案"

    以 MCP 工具搜索为例：在工具数量还很少时提前增加搜索式发现层，会引入索引、排序、召回率、缓存和
    权限过滤等新问题，还可能因为缺少真实使用数据而选错抽象。结果是 Context 未必节省多少，系统却已
    经增加了维护成本、延迟和新的失败路径。

    “工具 schema 已经明显占用 Context 预算”这样的触发标准，把决定建立在可观察的问题上。到那时，
    我们可以根据真实的工具数量、token 占用和调用模式设计方案，而不是只因为功能听起来有用就提前建
    设。
